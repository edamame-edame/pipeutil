# 親レビュアー提案書（独立版）

## 件名
Python 3.8 - 3.14 対応方針における CI 実行時間最適化案（現行提案書とは別枠）

## 位置づけ
本書は、既存の仕様提案・実装提案とは切り離した**親レビュアー視点の運用提案**である。
採否は実装仕様そのものではなく、CI 運用効率と品質保証バランスの観点で判断する。

## 背景
- プロジェクト基準を Python 3.8 - 3.14 に拡張したため、単純な全組み合わせ実行では CI コストが増加する。
- 一方で、最小版（3.8）と最新版（3.14）の健全性は継続監視が必要。
- 現行は Windows/Linux × 複数 Python で網羅度が高いが、PR ごとの待ち時間増加リスクがある。

## 提案の目的
- 品質を落とさずに、PR のフィードバック時間を短縮する。
- 失敗時の原因切り分け（OS起因か Python 版起因か）を維持する。
- 将来のバージョン追加時にも運用ルールを再利用できる形にする。

## 提案内容（2段階）

### 提案A: 常時実行マトリクスの圧縮（推奨）
PR では以下を常時実行する。

- Windows: Python 3.8, 3.14
- Linux: Python 3.8, 3.11, 3.14

意図:
- 最小版/最新版を両OSで担保
- 中間版代表（3.11）は Linux で保持
- 実行ジョブ数を抑制

### 提案B: 夜間フルマトリクス
スケジュール実行（nightly）で以下を実施する。

- Windows/Linux × Python 3.8, 3.11, 3.13, 3.14

意図:
- 互換性の網羅検証を夜間に集約
- PR は高速性重視、nightly は完全性重視に役割分担

## 期待効果
- PR サイクル短縮（レビュー待ち時間の削減）
- 互換性監視の継続（最小版・最新版の常時監視）
- 失敗分析の簡素化（常時系と夜間系で責務分離）

## リスクと対策
- リスク: PR 時点で 3.13 固有不具合を見逃す可能性
  - 対策: nightly で必ず検出、加えて release ブランチではフルマトリクス必須化

- リスク: マトリクス分離により運用が複雑化
  - 対策: workflow を `pr-fast` と `nightly-full` で明確分離し、README/CONTRIBUTING にルール明記

## 採用判定基準（レビュアー観点）
- PR の平均完了時間が現状より短縮されること
- nightly の失敗検出率が維持されること
- リリース前フルマトリクスで回帰がないこと

## 推奨アクション
1. まず提案Aを導入（1週間）
2. nightly の失敗傾向を確認
3. 問題なければ提案Bを正式運用化
4. 次期リリース前に KPI（時間/失敗率）で再評価
