## 1. 総評

提案書は「C++ をコアに据え、多言語はプロトコル互換ラッパーで広げる」という方向性自体は妥当で、導入コスト分離の方針も適切です。

一方で、現時点のまま Phase 1 に着手すると、**プロトコル定義の内部不整合**と**スコープ拡大**により、v1.0.0 で収束した互換性・非同期ライフサイクル・パッケージングの課題を再び広げるリスクがあります。v2.0.0 で本当に必要なのは「機能数の最大化」ではなく、**異言語相互接続の土台を崩さずに拡張可能な契約を先に固定すること**です。

---

## 2. 採用優先度の見直し

### 採用を強く推奨

- **L-002 プロトコル互換仕様 + 参照実装**
  - v2.0.0 の中核です。C# や将来の Java/Rust より先に、互換仕様・golden frame・エラー契約を固定する価値があります。
- **F-012 ハートビート / プロセス監視**
  - 長時間稼働サービスでは実運用価値が高く、プロトコル破壊なしでも設計しやすい候補です。

### 条件付きで採用

- **L-001 C# / .NET ラッパー**
  - ユースケースは強いですが、先に protocol freeze と interop test harness が必要です。
- **F-009 並行 RPC**
  - 実需はありますが、message_id 拡張だけでなく flow control, timeout, cancel, orphan response の契約が必須です。
- **F-010 ストリーミング**
  - 実需はありますが、まずは v1 互換で解ける範囲を切り分けるべきです。

### 現時点では core 採用を推奨しない

- **F-011 Pub/Sub メッセージルーター**
  - これは transport core ではなく broker 製品の責務です。別パッケージ化またはサンプル実装が妥当です。
- **L-003 Java ラッパー**
  - UDS と依存関係の整理が未確定で、v2.0.0 の必須機能とは言いにくいです。
- **L-004 Rust クレート**
  - 方向性はよいものの、今の priority ではありません。
- **F-014 TLS オーバーレイ**
  - 別パッケージ方針は適切です。v2.0.0 core の着手対象からは外すべきです。

## 3. 主な指摘

### High: ヘッダ v2 のビット配置が矛盾している

- F-010 は stream flags を `flags` に載せています。
- F-013 は `flags` 上位 3bit を priority に再利用しています。
- Section 6 は `flags` と別に `priority(3bit) | stream_flags(5bit)` の新しいバイトを定義しています。

この状態では C++/Python/C# で同じフレーム定義を実装できません。**最初に 1 つの canonical header map を決めること**が必須です。

### High: F-010 の根拠は wire format ではなく実装制約に見える

提案書の説明では、問題の中心は「固定 64 KB 受信バッファ」です。これは v1 の `payload_size` 制約ではなく、read loop / buffer growth policy の問題です。したがって、**いきなり 16→24 byte の breaking change を採る前に、v1 互換の large payload / chunked API を評価すべき**です。

### Medium: F-011 はライブラリの責務を超えやすい

Pub/Sub は topic registry, fan-out, ordering, replay, backpressure, dead subscriber cleanup を必要とし、transport library の責務から外れやすいです。既存 `MultiPipeServer` の上に別 package として載せる方が、コアの安定性を守れます。

### Medium: Java 提案は依存関係が軽くない

Java 11 に標準 UDS はありません。Java 11 を維持するなら外部ライブラリが必要で、導入コスト分離の方針と衝突します。Java を進めるなら **Java 17+ 固定**または**別パッケージ + 追加依存明示**が必要です。

### Medium: 並行 RPC / ストリーミングには backpressure 契約が不足している

v1.0.0 のレビューでは pending request cleanup, timeout propagation, async completion ordering が重要論点でした。v2 でも queue limit, sender throttle, cancellation, partial failure を仕様化しない限り、同種の不具合を再発させやすいです。

### Medium: protocol version は「変更」ではなく「追加」として扱うべき

ヘッダ構造の差異自体は、接続時に protocol version を識別できるなら成立します。ただし、互換性を維持する前提なら **v1 を置き換えるのではなく v2 を追加する** 方針を明文化すべきです。

また、version を先行追加すると、C++/Python/C# の各実装が未確定の moving target を同時に追うことになります。したがって、**新しい protocol version の公開は、一連の新機能の仕様と実装が固まりきった後に行うべき**です。

## 4. レビュアー推奨の実装順

1. `PROTOCOL_COMPAT.md` と golden frame corpus を先に固定する。
2. capability negotiation / feature bitmap を追加し、v1/v2 共存方針を固める。
3. v1 互換で解ける large payload / chunked read を先に検証する。
4. そのうえで C# interop を最初の多言語対象として実装する。
5. Pub/Sub や TLS は core ではなく別パッケージ評価に回す。

### protocol version に関する補足合意

- protocol version は既存版の**変更**ではなく、互換性維持のための**追加**として扱う。
- 既存 v1 実装は維持し、v2 は capability negotiation で識別可能にする。
- 新 version の公開時点は、F-009/F-010 など関連する一連の新機能の仕様・実装・相互接続試験が完了した後に置く。
- つまり、先に version を増やして実装を追従させるのではなく、**機能群を固めてから version を追加する** 順で進める。

## 5. 判定

- **v2.0.0 の方向性**: 妥当
- **現提案のまま Phase 1 実装着手**: 非推奨
- **実装着手条件**: protocol freeze、capability negotiation、flow-control 契約の先行確定
- **versioning 方針**: v1 維持のまま新 version を追加し、その公開は新機能群の完了後に行う
