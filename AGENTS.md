# 条件
**C++ のバージョン** ... C++20
**Python のバージョン** ... Python3.8 - 3.14
**レビュー結果の保存先** ... `プロジェクトルートディレクトリ\review\%Y%m%d%H%M%S.csv` と `プロジェクトルートディレクトリ\review\whole.md`

## エージェントの役割分担

- 主実装者: Claude
- レビュアー: GPT 系モデル（Code X / GPT-5.4 など）
- `CLAUDE.md` は実装者向けルール、`CODEX.md` はレビュアー向けルールとして扱う
- レビュー依頼時は「実装者の修正反映確認」と「新規リスク検出」の 2 点を GPT 系レビュアーに求める

## review ディレクトリの保持方針

- 詳細レビューを圧縮して最終総括を作成した後は、`review` 配下の CSV は最新の集約 CSV だけを残す。
- `review/whole.md` は時系列ログではなく、最新リリース時点の単一総括として維持する。
- ユーザー向けの恒久情報は `README.md`、実装者向けの恒久ルールは `CLAUDE.md` / `AGENTS.md` / `.github/*` に移し、`review` は最終判断の記録に留める。

## review/whole.md の運用ルール

### 実装者（CLAUDE）のルール
- 指摘原文（`**指摘**:` 行）・影響・根拠は**一切変更しない**。
- 対応内容は指摘文の直下に改行して `→ XXX` の形式で追記するのみとする。
  ```
  **指摘**: ○○が問題。
  → spec/XX.md の△△を修正しました。✅
  ```
- 指摘文の削除・上書きは禁止。

### レビュアー（CODEX）のルール
- **指摘原文を編集・修正できるのはレビュアーのみ**。
- レビュー依頼ごとに、実装者が追記した対応内容（`→` 行）と実際の仕様書・コードを照合し、修正が正しく反映されているかを確認する。
- 確認済みの指摘は詳細セクションから削除し、「修正確認済み項目（注意点のみ）」へ 1 行要約で集約する。
- 未修正の指摘（特に Medium 以上）は **指摘 / 影響 / 根拠 / 対応状況** を維持する。
- `review/whole.md` 更新時は最終更新時刻と最新レビュー CSV 名を同期する。
- 常にフレッシュな視点で新たな指摘がないか確認する。一般的な機能要件・ソフトウェア安全性が担保できていれば指摘件数 0 でも問題ない。
- 集約完了後は、レビュー結果のうち恒久ルールとして有用なものを `README.md`、`CLAUDE.md`、`.github/*` に移送してから review を縮約する。
